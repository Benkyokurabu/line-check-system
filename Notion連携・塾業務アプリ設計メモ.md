# Notion連携・塾業務アプリ設計メモ

最終更新: 2026-07-16

## 前提

- 塾全体の正本データは現時点では Notion。
- 生徒・保護者とのやり取りは基本的に LINE 公式アカウント 1 つで行っている。
- ネットワークHDDにも Excel、一太郎、PDF 等のファイルが多数ある。最初は中身をすべて取り込まず、必要になったものから索引化・参照する。
- 将来的にアプリの完成度が上がれば、Notion編集からアプリ編集へ移る可能性もある。

## 基本方針

- 初期段階では Notion を正本、Supabase をアプリ用同期DBとして扱う。
- アプリはまず「LINE対応」「生徒検索」「生徒ごとの経緯確認」をしやすくする業務フロントとして作る。
- 最初からNotion全体の本文・全レコードを読むのではなく、まず共有範囲のDB構造を棚卸しする。
- 秘密情報やアクセストークンはチャットに貼らない。ローカルファイルまたは管理画面で扱う。

## データ取得元メモ（2026-07-16更新）

現時点の整理では、Notionだけを正本にせず、項目ごとに取得元を分ける。

### フォルダ内のクラス一覧表を正とする

- 生徒ID / 学籍番号
- 生徒氏名
- 学年
- 担任
- 所属クラス
- 校舎
- 学校名

### 現行アプリ / LINE連携を正とする

- 生徒本人LINEアカウント
- 保護者LINEアカウント
- 兄弟姉妹情報: LINEの保護者表示名に、兄弟姉妹全員を入れる運用ルールで扱う。

生徒本人LINEアカウントと保護者LINEアカウントは、現状の登録状態を元に扱う。Notionから新規に持ってくる項目ではない。

### Notionから取得する

- 保護者氏名・続柄
- 電話番号
- 入塾日 / 退塾日 / 休塾期間
- 必要な対応履歴・連絡履歴など、Notion側にしかない情報

### 後で別経路から取得する

- メールアドレス: 別の場所にあるため、後で持ってくる。

### 将来的に追加する

- 注意事項 / 対応メモ: そのうちアプリ側に入れる。

### 今は不要 / 取り込まない

- 住所: 不要。
- 請求 / 契約関連: 別システムがあるため、今は不要。

## LINE欠席連絡の自動登録方針

LINEメッセージをAIが判断し、欠席・遅刻・振替希望などだった場合、Notionの所定DBへ書き込むところまで実装したい。

ただし本番運用では、AIが直接「確定情報」として登録するのではなく、まず「AI判定・未チェック」として登録する。

推奨フロー:

1. LINE受信
2. Supabaseに原文保存
3. AIが分類・抽出
4. Notionの欠席・連絡管理DBへ `AI判定・未チェック` として自動登録
5. 人間が内容を確認・修正
6. 人間が `チェック済み` にする
7. スタッフは `チェック済み` のビューを見る

## 欠席・連絡管理DBの設計方針

Notion DBは最初は1つにする。

```text
欠席・連絡管理
```

主なプロパティ案:

- 状態: `AI判定・未チェック` / `チェック済み` / `差し戻し` / `対応不要`
- スタッフ共有: チェックボックス
- 種別: `欠席` / `遅刻` / `振替希望` / `その他`
- 生徒: 生徒DBへのリレーション
- 保護者: 必要なら保護者DBへのリレーション
- 日付
- 授業・クラス
- 原文
- AI要約
- AI信頼度
- 注意理由
- 連絡元: `LINE` / `電話` / `口頭`
- LINEメッセージID
- 確認者
- 確認日時
- 作成日時

AIが書き込む時:

```text
状態 = AI判定・未チェック
スタッフ共有 = false
```

人間が確認した時:

```text
状態 = チェック済み
スタッフ共有 = true
確認者 = 確認した人
確認日時 = 今
```

## Notionビュー案

- AI確認待ち
  - フィルタ: `状態 = AI判定・未チェック`
- スタッフ共有用
  - フィルタ: `状態 = チェック済み` または `スタッフ共有 = true`
- 今日の欠席
  - フィルタ: `状態 = チェック済み` かつ `日付 = 今日` かつ `種別 = 欠席`
- 差し戻し・要修正
  - フィルタ: `状態 = 差し戻し`

## 注意点

- Notionのビュー分けは権限分離ではない。同じDBを見る権限がある人は、フィルタを外すと未チェック情報も見られる可能性がある。
- 「未チェック情報を絶対に見せたくない」場合は、Notionビューではなくアプリ側で権限管理する必要がある。
- 同じLINEメッセージから重複登録しないよう、Notion側にも `LINEメッセージID` を持たせる。
- 生徒が一意に特定できない場合、兄弟がいる場合、日付が曖昧な場合、授業が特定できない場合は `AI判定・未チェック` のまま人間確認必須にする。
- 将来的に自動確定を行う場合も、条件を限定する。

自動確定候補:

- LINEユーザーと生徒が一意に紐づいている
- 日付が明確
- 欠席/遅刻などの種別が明確
- 対象授業が一意
- 同日の重複登録がない
- AI信頼度が高い

それ以外は確認必須。
